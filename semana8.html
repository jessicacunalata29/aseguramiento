Semana 8

Pruebas Unitarias

Las pruebas unitarias permiten verificar el correcto funcionamiento de una función, método o clase de manera independiente. Su propósito es detectar errores desde las primeras etapas del desarrollo, facilitando el mantenimiento del software y mejorando su calidad.

Ejercicios - Diapositiva 3

Ejercicio 1

Una función llamada calcularPromedio() recibe tres notas y devuelve el promedio final del estudiante.

Pregunta: ¿Puede considerarse una unidad para realizar una prueba unitaria? Explique su respuesta.

Sí. Una función representa una unidad de código porque realiza una tarea específica y puede probarse de forma independiente sin necesidad de ejecutar el resto del sistema.

Ejercicio 2

Analice el siguiente escenario:

Una prueba necesita conectarse a Internet para obtener información antes de ejecutarse.

Pregunta: ¿Qué principio FIRST no se está cumpliendo?

Principio Respuesta
Independent No se cumple porque la prueba depende de un servicio externo (Internet), por lo que su resultado puede variar.

Ejercicio 3

Seleccione el principio FIRST que corresponde a cada situación.

Situación Principio
La prueba tarda menos de un segundo. Fast
Siempre produce el mismo resultado. Repeatable
El resultado es aprobado o fallido automáticamente. Self-Validating
Se desarrolla antes del código. Timely

Estos principios garantizan que las pruebas unitarias sean rápidas, confiables y fáciles de mantener.

Ejercicios - Diapositiva 4 (Patrón AAA)

Ejercicio 1

Enunciado:

Observe el siguiente caso de prueba e identifique las partes del patrón AAA. 

double precio = 300;
double descuento = 0.20;
Calculadora calc = new Calculadora();

double resultado = calc.descuento(precio, descuento);

assertEquals(240, resultado);
Patrón Respuesta
Arrange Se crean las variables precio, descuento y el objeto Calculadora.
Act Se ejecuta el método calc.descuento().
Assert Se verifica que el resultado obtenido sea igual a 240.

Ejercicio 2

Enunciado:

Ordene correctamente las etapas del patrón AAA.

  • Verificar el resultado.
  • Preparar los datos.
  • Ejecutar la función.
Orden Correcto
1. Arrange → Preparar los datos.
2. Act → Ejecutar la función.
3. Assert → Verificar el resultado.

Ejercicio 3

Enunciado:

Una prueba crea un objeto Usuario, registra la información y finalmente verifica que el nombre almacenado sea correcto.

Etapa Actividad
Arrange Crear el objeto Usuario y asignar los datos.
Act Ejecutar el método registrar().
Assert Comprobar que el nombre registrado sea correcto.

Ejercicios - Diapositiva 5 (Diseño de Casos de Prueba)

Ejercicio 1

Una función registra la cantidad de productos en inventario.

La cantidad válida debe estar entre 1 y 500.

Tipo de Caso Entrada Resultado Esperado
Válido 250 Registro correcto.
Límite 1 y 500 Aceptados correctamente.
Inválido -5 y 700 Mostrar mensaje de error.

Ejercicio 2

Una aplicación bancaria permite retirar dinero únicamente entre $10 y $1000.

Tipo Valor Resultado
Válido $500 Retiro exitoso.
Límite $10 y $1000 Retiro permitido.
Inválido $5 y $1500 Operación rechazada.

Ejercicio 3

Una plataforma educativa acepta calificaciones entre 0 y 100.

Tipo Entrada Resultado Esperado
Válido 80 Nota registrada.
Límite 0 y 100 Nota aceptada.
Inválido -10 y 120 Error de validación.

Conclusión

Las pruebas unitarias constituyen la base del aseguramiento de la calidad del software, ya que permiten verificar el funcionamiento correcto de cada unidad de código antes de integrar el sistema completo. La aplicación de los principios FIRST, el uso del patrón AAA y el diseño adecuado de casos de prueba ayudan a detectar errores de manera temprana, reducir costos de mantenimiento y mejorar la confiabilidad del software.
