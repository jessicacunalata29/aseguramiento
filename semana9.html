IEEE 829

Estándar para la Documentación de Pruebas de Software

¿Qué es el IEEE 829?

El IEEE 829 es un estándar internacional desarrollado por el Institute of Electrical and Electronics Engineers (IEEE), cuyo propósito es establecer una estructura organizada para la documentación de las pruebas de software. Este estándar ayuda a que las actividades de testing sean planificadas, ejecutadas y documentadas de manera uniforme, facilitando la comunicación entre desarrolladores, analistas, testers y clientes.

Su principal objetivo es garantizar que todas las pruebas realizadas durante el desarrollo de un sistema sean registradas correctamente, permitiendo mantener la trazabilidad, la calidad y el control del proceso de pruebas.

Historia

El estándar IEEE 829 fue publicado por primera vez en el año 1983 y recibió actualizaciones posteriores para adaptarse a las nuevas metodologías de desarrollo de software. Aunque actualmente gran parte de su contenido fue integrado en la norma internacional ISO/IEC/IEEE 29119, IEEE 829 sigue siendo uno de los modelos más utilizados para enseñar documentación de pruebas de software en universidades y proyectos empresariales.

Objetivos del IEEE 829

📋 Organización

Define una estructura clara para documentar todas las actividades de pruebas.

✅ Calidad

Permite verificar que el software cumpla con los requisitos establecidos.

📊 Seguimiento

Facilita el control de las pruebas y el seguimiento de los defectos encontrados.

🤝 Comunicación

Mejora la comunicación entre el equipo de desarrollo, QA y clientes.

Importancia del IEEE 829

Implementar este estándar ofrece múltiples ventajas dentro del ciclo de vida del desarrollo de software, ya que permite organizar la información de las pruebas, reducir errores de documentación y proporcionar evidencia del proceso de aseguramiento de calidad.

  • Documentación clara y organizada.
  • Mayor trazabilidad entre requisitos y pruebas.
  • Facilita auditorías de calidad.
  • Permite controlar el avance del testing.
  • Reduce errores durante la ejecución de pruebas.
  • Facilita la gestión de defectos.
  • Ayuda en la toma de decisiones antes de liberar el software.

Documentos definidos por IEEE 829

Documento Descripción
Plan de Pruebas Define la planificación general del proceso de pruebas.
Diseño de Pruebas Describe las condiciones y estrategia de prueba.
Casos de Prueba Contienen los pasos, entradas y resultados esperados.
Procedimientos de Prueba Indican cómo ejecutar cada caso de prueba.
Registro de Ejecución Guarda los resultados obtenidos durante las pruebas.
Reporte de Defectos Documenta los errores encontrados durante la ejecución.
Reporte Final Resume el estado final del proceso de pruebas.

Ejemplo de Caso de Prueba (Estado Rechazado)

Un caso de prueba es un conjunto de condiciones, datos y pasos que permiten verificar si una funcionalidad del sistema cumple con el comportamiento esperado. Durante su ejecución, el resultado puede ser aprobado (PASS) o rechazado (FAIL). En este ejemplo se presenta un caso rechazado debido a un error crítico de autenticación.

Campo Descripción
ID del Caso TC-LOGIN-001
Módulo Inicio de Sesión
Nombre del Test Validar el inicio de sesión utilizando una contraseña incorrecta.
Prerrequisito El usuario debe estar registrado en el sistema con un correo electrónico válido.
Datos de Entrada Correo: victor@gmail.com
Contraseña: 12345678 (incorrecta)
Pasos de Ejecución 1. Abrir la aplicación SmartTooth.
2. Ingresar el correo electrónico registrado.
3. Escribir una contraseña incorrecta.
4. Presionar el botón Iniciar Sesión.
Resultado Esperado El sistema debe impedir el acceso y mostrar el mensaje:

"Credenciales incorrectas"

Sin permitir el ingreso al sistema.
Resultado Obtenido El sistema permitió el acceso aun cuando la contraseña era incorrecta y redirigió al usuario hacia la pantalla principal.
Estado ❌ RECHAZADO (FAILED)
Motivo del Rechazo El sistema presenta un defecto de autenticación, ya que permite iniciar sesión utilizando credenciales inválidas. Este comportamiento representa un riesgo crítico de seguridad y debe corregirse antes de la liberación del sistema.

Reporte de Defecto

Cuando un caso de prueba falla, el defecto debe documentarse correctamente para que el equipo de desarrollo pueda reproducirlo, corregirlo y verificar posteriormente la solución.

Campo Valor
ID Defecto BUG-0001
Título El sistema permite iniciar sesión con una contraseña incorrecta.
Módulo Inicio de Sesión
Severidad 🔴 CRÍTICO
Prioridad P1 — Inmediata
Pasos para Reproducir 1. Abrir SmartTooth.
2. Ingresar el correo registrado.
3. Escribir una contraseña incorrecta.
4. Hacer clic en Iniciar Sesión.
5. Observar el comportamiento del sistema.
Resultado Esperado El sistema debe rechazar el acceso mostrando el mensaje:

"Credenciales incorrectas"
