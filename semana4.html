Investigacion

1. Defect Density (Densidad de Defectos)

Es una métrica que mide la cantidad de errores encontrados en relación con el tamaño del software. Se utiliza para evaluar la calidad del producto desarrollado.

Ejemplo

Un sistema de citas médicas tiene 15 defectos encontrados en 1.500 líneas de código.

Densidad de Defectos = 15 / 1500 = 0.01

Esto significa que existe aproximadamente 1 defecto por cada 100 líneas de código.

2. Cobertura de Pruebas

Es el porcentaje del código que ha sido ejecutado durante las pruebas de software. Permite conocer qué tan probado está el sistema.

Ejemplo

Un proyecto tiene 1.000 líneas de código y las pruebas ejecutan 850 líneas.

Cobertura = (850 / 1000) × 100 = 85%

El sistema posee una cobertura de pruebas del 85%.

3. Complejidad Ciclomática

Es una métrica que mide la cantidad de caminos lógicos independientes que existen dentro de un programa. Ayuda a determinar qué tan complejo es el código.

Ejemplo

Un programa para registrar pacientes tiene tres decisiones:

  • Verificar usuario.
  • Validar datos ingresados.
  • Confirmar la cita médica.

Mientras más decisiones tenga el programa, mayor será la complejidad ciclomática y más pruebas se necesitarán.

4. Acoplamiento entre Clases

Es el grado de dependencia que existe entre dos o más clases de un sistema. Un bajo acoplamiento facilita el mantenimiento y la reutilización del código.

Ejemplo

En un sistema de citas médicas existe una clase Paciente y una clase Cita. La clase Cita utiliza información del Paciente para registrar una consulta.

Si la clase Cita depende demasiado de la clase Paciente, existe un alto acoplamiento. Si cada clase funciona de manera más independiente, el acoplamiento es bajo y el sistema será más fácil de mantener.

5. ¿Qué es un Grafo?

Un grafo es una estructura formada por nodos y conexiones llamadas aristas. En ingeniería de software se utiliza para representar flujos de procesos, navegación y análisis de complejidad.

6. Operación del Producto

Se refiere al comportamiento del software durante su uso diario por parte de los usuarios. Incluye eficiencia, seguridad, confiabilidad y facilidad de uso.

7. Revisión del Producto

Proceso de evaluación del software para identificar errores, defectos y oportunidades de mejora antes de su entrega.

8. Transición del Producto

Es el proceso mediante el cual un sistema pasa desde el desarrollo hacia el entorno real de operación.

9. Portabilidad

Capacidad del software para ejecutarse correctamente en diferentes plataformas, sistemas operativos o dispositivos.

Ejemplo: Una aplicación que funciona tanto en Windows como en Linux.

10. Reutilización

Capacidad de utilizar componentes de software ya desarrollados en nuevos proyectos para reducir tiempo y costos.

11. Interoperabilidad

Capacidad de distintos sistemas o aplicaciones para intercambiar información y trabajar conjuntamente.

Ejemplo: Un sistema de citas médicas conectado con un sistema de facturación.

12. Modelo CMMI

Capability Maturity Model Integration es un modelo que permite mejorar y evaluar los procesos de desarrollo de software dentro de una organización.

13. Nivel 1 - Inicial

Los procesos son impredecibles y dependen principalmente de los esfuerzos individuales.

14. Nivel 2 - Gestionado

Los procesos son planificados, monitoreados y controlados para cumplir objetivos específicos.

15. Nivel 3 - Definido

Los procesos están documentados y estandarizados en toda la organización.

16. Nivel 4 - Gestionado Cuantitativamente

La organización utiliza métricas y datos estadísticos para controlar y mejorar sus procesos.

17. Usabilidad General

Capacidad del software para ser comprendido, aprendido y utilizado fácilmente por los usuarios.

  • Facilidad de aprendizaje.
  • Facilidad de uso.
  • Satisfacción del usuario.

18. Mantenibilidad

Capacidad del software para ser modificado, corregido o mejorado de manera eficiente.

19. Portabilidad (ISO 25010)

Permite trasladar el software a distintos entornos sin realizar grandes modificaciones.

20. Independencia del Dispositivo

Capacidad de una aplicación para funcionar correctamente en diferentes dispositivos como computadoras, tablets y teléfonos móviles.

21. Auto-contenido

Característica de un componente de software que puede funcionar de forma independiente sin depender excesivamente de otros módulos.

Conclusión

El aseguramiento de calidad permite evaluar la calidad del software mediante métricas, estándares y modelos como CMMI. Conceptos como densidad de defectos, cobertura de pruebas, complejidad ciclomática, interoperabilidad y portabilidad ayudan a desarrollar sistemas más confiables, mantenibles y eficientes para los usuarios.
