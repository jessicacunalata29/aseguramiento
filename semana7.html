Tipos de Pruebas de Software

Las pruebas de software son actividades cuyo objetivo es verificar que un sistema funcione correctamente, cumpla con los requisitos establecidos y ofrezca un producto de calidad al usuario final. Estas pruebas permiten detectar errores antes de que el software sea entregado, reduciendo costos de mantenimiento y aumentando la confiabilidad del sistema.

1. Desarrollo de Conceptos (Diapositiva 3)

Pruebas Funcionales

Las pruebas funcionales verifican que cada función del sistema cumpla con los requisitos establecidos por el cliente. Se enfocan en comprobar qué hace el sistema sin importar cómo está desarrollado internamente. Generalmente se evalúan funciones como el inicio de sesión, el registro de usuarios, cálculos, consultas, reportes y cualquier proceso que forme parte de los requerimientos funcionales.

Su objetivo principal es garantizar que todas las funcionalidades produzcan los resultados esperados cuando reciben datos válidos e inválidos.

Pruebas No Funcionales

Las pruebas no funcionales evalúan la calidad del sistema y su comportamiento durante la ejecución. No verifican una función específica, sino características como el rendimiento, la velocidad, la seguridad, la disponibilidad, la usabilidad y la compatibilidad.

Estas pruebas ayudan a garantizar que el software pueda utilizarse correctamente incluso cuando existe una gran cantidad de usuarios o condiciones extremas.

Pruebas Estáticas

Las pruebas estáticas se realizan sin ejecutar el programa. Consisten en revisar documentos, diagramas, requisitos, código fuente y diseño para detectar errores antes de que el sistema sea compilado o ejecutado.

Entre las actividades más comunes están las inspecciones, revisiones técnicas, walkthroughs y análisis estático del código.

Pruebas Dinámicas

Las pruebas dinámicas consisten en ejecutar el software para comprobar su funcionamiento utilizando diferentes datos de entrada y analizando las respuestas obtenidas. Permiten identificar errores de lógica, fallos en procesos y problemas de integración.

Estas pruebas representan la mayor parte del proceso de aseguramiento de calidad debido a que verifican el comportamiento real del sistema.

3. Ejercicios Adicionales (Diapositiva 10)

Ejercicio 1

Un desarrollador crea una función que calcula el total de una factura y verifica que el resultado sea correcto para diferentes valores.

Tipo de prueba: Prueba Unitaria

Tipo de caja: Caja Blanca

Se trata de una prueba unitaria porque evalúa una función específica del sistema y el desarrollador conoce el código fuente.

Ejercicio 2

Un usuario intenta iniciar sesión con credenciales válidas e inválidas para comprobar si el sistema responde correctamente.

Tipo de prueba: Prueba Funcional

Tipo de caja: Caja Negra

El usuario únicamente verifica el comportamiento del sistema sin conocer cómo está implementado internamente.

Ejercicio 3

Después de actualizar el módulo de inventario, el equipo ejecuta nuevamente todas las pruebas realizadas anteriormente para asegurar que el resto del sistema continúe funcionando correctamente.

Tipo de prueba: Prueba de Regresión

Tipo de caja: Caja Negra

Esta prueba confirma que los cambios realizados no afectaron funcionalidades existentes.

Ejercicio 4

Un especialista en seguridad intenta acceder a la base de datos mediante ataques de inyección SQL para verificar que el sistema sea seguro.

Tipo de prueba: Prueba de Seguridad

Tipo de caja: Caja Gris

El especialista posee conocimiento parcial de la arquitectura para detectar vulnerabilidades.

4. Estrategia de Pruebas para un Sistema de Inventario

Un sistema de inventario administra productos, proveedores, entradas, salidas y reportes. Para garantizar su correcto funcionamiento es necesario aplicar diferentes tipos de pruebas durante todo el ciclo de desarrollo.

Módulo Tipo de prueba Tipo de caja Herramienta Criterio de éxito
Inicio de sesión Prueba Funcional Caja Negra Selenium Permite ingresar únicamente con credenciales válidas.
Registro de Productos Prueba Unitaria Caja Blanca JUnit Los datos se almacenan correctamente en la base de datos.
Actualización del Stock Prueba de Integración Caja Gris Postman El stock se actualiza correctamente después de cada movimiento.
Búsqueda de Productos Prueba de Sistema Caja Negra Cypress Las búsquedas muestran resultados correctos en menos de dos segundos.
Reportes Prueba Funcional Caja Negra Selenium Los reportes contienen información completa y precisa.
Rendimiento General Prueba de Rendimiento Caja Negra JMeter El sistema soporta al menos 500 usuarios simultáneos sin errores.
Seguridad Prueba de Seguridad Caja Gris OWASP ZAP No existen vulnerabilidades críticas.
Entrega Final Prueba de Aceptación (UAT) Caja Negra Pruebas Manuales El cliente aprueba el funcionamiento del sistema.

Importancia de la estrategia

Implementar una estrategia de pruebas permite detectar errores desde las primeras etapas del desarrollo, mejorar la calidad del software y garantizar que el sistema de inventario funcione correctamente antes de ser entregado al cliente. Además, reduce costos de mantenimiento, incrementa la confiabilidad del sistema y mejora la experiencia de los usuarios finales.

Conclusión

Las pruebas de software constituyen una de las actividades más importantes dentro del aseguramiento de la calidad. Aplicar correctamente pruebas funcionales, no funcionales, unitarias, de integración, sistema, aceptación, regresión, humo y seguridad permite desarrollar aplicaciones más confiables, seguras y eficientes. Una adecuada estrategia de pruebas ayuda a prevenir errores, optimizar el mantenimiento del sistema y garantizar el cumplimiento de los requisitos del cliente.
